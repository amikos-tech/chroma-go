> **Revision note (Round 2 review feedback):**
> - **[FROM CODEX HIGH]** Logger noop fallback gap: `APIClientV2` (state client) defaults to `NoopLogger`, so `localDeleteCollectionFromCache` checking `client.logger != nil` would always be true and never fall back to stderr. Resolution: `embeddedLocalClient.logger` is nil by default (NOT NoopLogger). The nil check `if client.logger != nil` on `embeddedLocalClient` correctly triggers stderr fallback when unset. For the state client path (`localDeleteCollectionFromCache`), `WithPersistentLogger` propagates the logger via `WithLogger(l)` which replaces the default NoopLogger with the real logger -- so structured logging works there too. When no `WithPersistentLogger` is set, state client keeps its NoopLogger default and `localDeleteCollectionFromCache` logs via `client.logger.Error(...)` which is a no-op. This is acceptable because `deleteCollectionState` (on embeddedLocalClient) already logs the same error to stderr.
> - **[FROM CODEX MEDIUM]** WithPersistentLogger vs WithLogger naming: Explicitly stated that `WithPersistentLogger` is chosen to avoid clashing with existing `WithLogger(ClientOption)` API. `WithPersistentLogger` sets BOTH embedded client logger AND state client logger in one call.
> - **[FROM CODEX MEDIUM]** Tests should exercise real constructor path: Added `TestWithPersistentLogger_RealConstructorPath` that applies options to a real `localClientConfig` and verifies the full propagation chain, not just manual struct setting.
> - **[FROM CODEX LOW]** Dual Sync() note: Same logger attached to both embedded and state client means Sync() could be called twice. Acceptable -- loggers are designed to be Sync-safe.

**CRITICAL DESIGN NOTE (addresses Codex HIGH -- logger noop fallback gap):**

The `embeddedLocalClient.logger` field is nil by default (NOT `NoopLogger`). This is intentional and different from `BaseAPIClient.logger` which defaults to `NoopLogger`. The nil check pattern `if client.logger != nil { ... } else { logToStderr(...) }` ensures:
- When `WithPersistentLogger` is set: errors go to the structured logger
- When NOT set: errors go to stderr (preserving current behavior)

If we defaulted to `NoopLogger`, the stderr fallback would never trigger and errors would be silently swallowed. The `APIClientV2` (state client) keeps its `NoopLogger` default -- when `WithPersistentLogger` is NOT set, `localDeleteCollectionFromCache` errors go through the noop logger (effectively silent). This is acceptable because `deleteCollectionState` on the `embeddedLocalClient` already logs the same EF close error to stderr.

**2. Add WithPersistentLogger option (client_local.go):**

Add after the existing `WithPersistentClientOptions` function:

```go
// WithPersistentLogger sets a structured logger for the persistent client.
// Auto-wire and close errors route through the logger instead of stderr.
// The logger propagates to both the embedded client and the internal state
// client so that localDeleteCollectionFromCache errors are also captured.
func WithPersistentLogger(l logger.Logger) PersistentClientOption {
    return func(cfg *localClientConfig) error {
        cfg.logger = l
        cfg.clientOptions = append(cfg.clientOptions, WithLogger(l))
        return nil
    }
}
```
