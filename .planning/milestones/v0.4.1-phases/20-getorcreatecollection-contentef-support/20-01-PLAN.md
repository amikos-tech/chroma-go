```go
// SetContentEmbeddingFunction persists content EF config by delegating to SetEmbeddingFunction
// when the content EF also implements EmbeddingFunction.
// Nil-safe: no-op when ef is nil.
// Best-effort: no-op when ef does not implement EmbeddingFunction (content-only EFs skip persistence).
func (c *CollectionConfigurationImpl) SetContentEmbeddingFunction(ef embeddings.ContentEmbeddingFunction) {
    if ef == nil {
        return
    }
    denseEF, ok := ef.(embeddings.EmbeddingFunction)
    if !ok {
        return
    }
    c.SetEmbeddingFunction(denseEF)
}
```

Note: Schema does NOT have SetContentEmbeddingFunction. Only CollectionConfigurationImpl does.
For Schema path, type-assert contentEF to EmbeddingFunction and call Schema.SetEmbeddingFunction.

IMPORTANT — Config persistence semantics:
- SetContentEmbeddingFunction and SetEmbeddingFunction both write to the SAME persisted #embedding config slot.
- When both denseEF and contentEF are provided, the contentEF persistence block runs AFTER the denseEF block.
- If contentEF also implements EmbeddingFunction, it overwrites the denseEF config. This is correct: an explicit contentEF that also supports dense should take precedence over a default ORT fallback.
- ContentEFs that do NOT implement EmbeddingFunction will have config persistence silently skipped. They still wire correctly at runtime; only server-side config storage is affected.
- WithDisableEFConfigStorage skips ALL config persistence (both denseEF and contentEF), because the early return at the disableEFConfigStorage check happens before both blocks.
